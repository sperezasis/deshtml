GitHub Structure

Two types of repos.
Code repos hold code. Team repos are your team's home.

Every product has code repos for source code. Every team has a team repo for everything else — issues, templates, processes, automations, and documentation. Your team repo is your operational home — one URL for everything that isn't code.

▾

The two kinds

Team Repo
Where work is planned and tracked
  • All team issues: tasks, projects, milestones
  • Issue templates, automations, GitHub Actions
  • Team docs: README, onboarding, runbooks
  • Labels specific to the team's needs
Code Repo
Where code lives — nothing else
  • Source code, branches, pull requests
  • CI/CD pipelines, releases, version tags
  • No issue tracking whatsoever
  • PRs reference team repo issues cross-repo

The rule is absolute. All issues live in team repos. Code repos have code only. A developer fixing a MemberPress bug creates the issue in membercore-dev with label MemberPress. The PR in the code repo references it: Fixes membercore-dev#47.

A team repo contains everything a team needs to operate, in one place.

Issue Definition

Every issue follows one format.
Repo → Type → Labels.

Every issue is defined by three things: the repo it lives in (who owns it), its type (what kind of work), and its labels (which specific kind). Any person on any team can read any issue and understand it instantly.

▾

Three components

Every issue is defined by three things. Together, they tell you everything you need to know at a glance.

Who
Repo
The issue lives in the repo of the team that does the work. One issue, one owner.If two teams are involved, each gets their own issue in their own repo — connected as sub-issues of a shared parent. The repo is about who DOES the work, not who reports it.
What
Type
What kind of work is this? PROJECT, MILESTONE, TASK. Every issue gets exactly one type. Same vocabulary across every team.
Which
Labels
Which specific kind? Bug, enhancement, MemberPress, MC Cart... Each team defines labels that fit their work. As many as needed.

In practice

Repo + Type + Labels = the full picture
membercore-dev + TASK + bug MemberPress = MC Dev team, task, MemberPress bug.
innovations + PROJECT + rock AI Co-Pilot = Innovations team, project, quarterly rock, AI Co-Pilot.
design + TASK + MC Cart = Design team, task, MC Cart project.

Issue types

The type tells you what structural level a piece of work represents. It's the same vocabulary across every team — when you see a type, you know exactly what you're looking at.

3
types
PROJECT — coordinated work with phases and sub-issues
MILESTONE — a phase or group within a project
TASK — any individual piece of work
All 3 types — expand for full reference
TypeWhat it means
PROJECT A body of work with phases and sub-issues. Multi-task initiative that needs coordination.
MILESTONE A phase or group within a project. Groups related tasks into a trackable unit.
TASK Any individual piece of work — the default type. The repoThe repo tells you which team does the work. A TASK in the design repo is design work. A TASK in membercore-dev is dev work. The type doesn't need to repeat what the repo already says. tells you which team does it. Labels tell you which kind.

Labels

Types are org-wide — the same 3 everywhere. Labels are the opposite: each team defines their own. A label set reflects how a team thinks about its work. There's no org-wide label list and no approval process — each team owns theirs.

Types
Org-wide. One per issue.
  • Same 3 types across every team
  • Exclusive — an issue has exactly one type
  • Answers: what kind of work is this?
Labels
Team-scoped. As many as needed.
  • Each team defines labels that fit their work
  • Additive — stack as many as apply
  • Answers: which product, which kind, which project?

Projects & Tasks

All work lives on one spectrum.
From a single task to a company-wide initiative — one escalation path.

A bug fix, a feature, a product launch — all work starts as an issue. As complexity grows, you add structure: sub-issues to organize, milestones to phase, sub-projects to divide. You never jump to a different system — you scale up within the same one.

▾

Five levels of complexity

Every piece of work fits somewhere on this spectrum. The first two levels are informal — you're just organizing your own work. From "Small project" onward, the work gets type PROJECT, a pitch, and formal coordination.

Level 1
Single task
One issue, one assignee. A bug, a request, a quick fix.
TASK Fix checkout timeout
Level 2
Complex task
Too big for one issue — break it into sub-issues. Same type, no formality. You're organizing your own work.
TASK Refactor payment flow
  ├ TASK Extract gateway logic
  ├ TASK Add retry handling
  └ TASK Write tests

The line. Above this point, you're organizing your own work — sub-issues are free, no type change, no process. Below this point, work involves multiple people or teams and needs formal coordination: type PROJECT, a pitch, and tracking.

Level 3 — Small project
Project + Tasks
Multiple people, one team. A parent issue with type PROJECT and a few tasks underneath.
PROJECT Add dark mode
  ├ TASK Build toggle
  ├ TASK Dark palette
  └ TASK Update docs
Level 4 — Medium project
Project + Milestones + Tasks
Work has phases. Milestones group tasks into stages — Design, Dev, QA — each trackable separately.
PROJECT WLMS Plugin
  ├ MILESTONE Design
  │  └ TASK Mockups
  ├ MILESTONE Dev
  │  └ TASK Build API
  └ MILESTONE QA
     └ TASK Pass 1
Level 5 — Large project
Project + Sub-projects + Milestones + Tasks
Multiple workstreams, often across teams. Each sub-project has its own milestones and tasks.
PROJECT MC Cart
  ├ PROJECT Core Checkout
  │  ├ MILESTONE Design
  │  │  └ TASK ...
  │  └ MILESTONE Dev
  │     └ TASK ...
  └ PROJECT Multi-Currency
     └ ...

Work top to bottom. The first "yes" wins.

If you're unsure, start lower. You can always promote a task to a project or add milestones later.

How to create a project

Every project (Level 3+) has two documents and a folder in the project-management repo. This is what makes a project official — without these, it's just an issue with sub-issues.

Document 1
Pitch
Problem, proposed solution, scope, dependencies, open questions, rough estimate. Written before work begins. This is how the project gets approved.
Document 2
Technical Brief
Architecture, code structure, DB schema, key decisions, repos involved. Written when development starts. This is the reference for the team building it.

Where they live. Both documents are Markdown files in the project-management repo, inside the teams/ folder, organized by team and sub-team.

Folder structure — project-management repo

Teams with sub-teams (MemberCore, IPJ) have one folder per sub-team. Teams without sub-teams hold projects directly. Templates live in templates/PITCH-TEMPLATE.md and templates/TECHNICAL-BRIEF-TEMPLATE.md.

The parent issue in GitHub links to these documents in its Key Docs section. If a project has sub-projects, each gets its own Pitch and Technical Brief inside a subfolder.

Start simple, scale up. If you're unsure whether something is a task or a project, start with sub-issues. You can always promote to a project later — add the type, write the pitch, create the folder. The system grows with the work.

Team Boards

Every team has a board. In GitHub, boards are called Projects.
The repo stores issues. The board is where they're organized, filtered, and tracked.

A boardA GitHub Project is the management layer on top of a team repo. The repo holds the issues. The board shows them — filtered, sorted, grouped, tracked — tailored to each team's workflow. One is the filing cabinet, the other is the desk.

▾

Repo vs. board

Two different things in GitHub. Both essential, different purposes.

Team Repo
Where issues live
  • Issues, templates, labels, automations
  • The source of truth — every issue starts here
  • One repo per team (or sub-team)
Board
How they're managed
  • Views, filters, sorting, custom fields
  • The dashboard — track progress, spot blockers
  • One board per team (some sub-teams share)

Issues arrive automatically

Every issue created in a team repo appears on the team's board the moment it's created — GitHub Actions handle this. No manual steps, no dragging issues between places.

Each team configures their board

Views, fields, and workflows are defined by each team to match how they work. A dev team's board looks different from a design team's board — different work, different needs, different setup.

Views
Multiple ways to see the same data
Table — spreadsheet rows, sort by any field. Board — kanban columns by status. Roadmap — timeline with dates. Hierarchy — parent/child nesting.
Fields
Each team tracks what matters
Status, health, priority, dates, quarters — each team defines the fields that fit their workflow. Up to 50 per board.
Automation
Zero maintenance
Status auto-updates when PRs merge. Done items auto-archive after a period. The board maintains itself.
Recommended views

Teams are free to create whatever views they need. These four are a solid starting point.

Day-to-day
By Status
Kanban board grouped by status columns. The default view — backlog, in progress, blocked, and done at a glance.

My Tasks

Cross-Team

Cross-team work follows one path.
Need something from another team? Open an issue in their repo.

When Team A needs something from Team B, there's one process: open the issue in Team B's repo, with the right type and labels. The parent issue stays with the requesting team, sub-issues fan out to whoever does the work.

▾

One process

Every cross-team request follows the same three steps. No Slack threads, no meetings to coordinate handoffs — one path that works the same way every time.

Step 1
Open a sub-issue in their repo
From your parent issue, create a sub-issue directly in the other team's repo.
Step 2
It appears on their board
Automatically. No manual steps, no notifications to send.
Step 3
They work it, you track it
They see a task on their board. You see progress on yours.

The requesting team keeps ownership of the project. The receiving team sees the task on their board like any other piece of work — same labels, same workflow, same process. Nobody changes how they work.
